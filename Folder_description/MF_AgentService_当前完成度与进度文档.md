### 4.2 受控知识同步与事件消费

已新增内部接口：

```text
POST /api/agent/knowledge/internal-sync
```

该接口使用 `X-MF-Internal-Token` 和环境变量 `MF_AGENT_KNOWLEDGE_SYNC_KEY` 鉴权，作为人工补偿和排障入口。正式主流程改为 AgentService 使用 `MF_EP_INTERNAL_TOKEN` 按默认 30 秒间隔拉取 MF_EP 的待确认同步事件。

事件消费支持：

- 拉取 `publish`、`offline`、`rollback` 事件；
- 刷新关键词索引，向量索引已启用时尽力更新但不阻塞关键词索引；
- 发布/回滚后验证内容进入索引，下线后验证内容不再命中；
- 验证成功后确认 MF_EP 事件；
- 失败时将状态、失败次数和原因回写 MF_EP 事件表；未达到阈值时保持 `pending` 并在下次轮询重试，达到阈值或等待超时后标记为 `failed` 或 `timed_out`，仅由“MF_EP 人工恢复 -> AgentService 内部重试”顺序再次处理。

人工接口支持：

- 以 `requestId` 保证成功请求幂等；
- 关联本次变更的一个或多个内容 ID；
- 立即执行轻量全量知识刷新，覆盖发布与下线；
- 返回成功状态、索引文档数量、失败原因和完成时间；
- 刷新失败时允许使用相同 `requestId` 重试。

当前 EP 侧公开知识读取使用列表接口，因此事件中的内容 ID 用于验证和审计关联，实际执行为全量轻量刷新。数据中台应保存 MF_EP 事件 ID、候选/草稿/正式内容 ID、同步状态和确认时间，作为发布审计。详细契约见：

`F:\20260518-xiangmu\MF_Project\Folder_description\MF_AgentService_知识同步接口对接说明.md`

生产环境配置位于 `src/main/resources/application-prod.yml`，地址、令牌、轮询间隔、重试上限和超时均从环境变量读取，不提交任何密钥值。
