# MF_EP 客服 40 类问题流程与数据契约任务书

更新时间：2026-07-14  
责任方：MF_EP 负责人  
协作方：MF_AgentService 负责人、MF_DataCenter 负责人  
优先级：P0，先完成规则与数据契约定义，不在本任务中接入或训练模型

## 1. 目标与边界

本任务的目标是把 MF_EP 的客服业务规则从零散代码和口头约定中整理为可审核、可实现的 40 类流程。每一类都必须明确：查询哪些真实数据、筛选与排序规则、用户权限、缺信息时的追问、无结果时的兜底文案。

MF_EP 负责业务事实、商品/订单/百科接口与权限规则；MF_AgentService 负责理解自然语言、调用已确认的接口和组织回答；MF_DataCenter 负责统计未解决问题与低满意度反馈。

本任务不包括：模型训练、模型 API Key、提示词调优、直接修改 AgentService 回答逻辑。

## 2. 交付要求

1. 在本文件的 40 类清单中补全每类的字段、规则和文案，并标记 `已确认`、`待补数据` 或 `暂不支持`。
2. 对无法由现有接口满足的规则，单列“MF_EP 接口/字段缺口”，不得让 AgentService 猜测数据。
3. 为每一类提供至少 3 条真实用户问法和 1 条标准成功回答、1 条缺信息追问、1 条无结果兜底。
4. 将最终确认的规则整理为结构化 YAML 草案，供 AgentService 后续读取；YAML 不保存密钥、账号、密码或真实订单数据。

建议的 YAML 单条结构：

```yaml
id: product_recommendation
intent: product_recommendation
examples: ["推荐苹果树商品"]
required_slots: [crop_name]
query: { source: mf_ep_product, filters: { status: on_sale }, sort: [sales_count_desc], tie_breaker: random_one }
permissions: { login_required: false }
response: { missing_slot: "...", empty: "...", success: "..." }
```

## 3. 40 类问题清单

| ID | 问题类型 | MF_EP 必须确认的数据/接口 | 必须确认的业务规则 | 缺信息或无结果时的处理 |
| --- | --- | --- | --- | --- |
| 01 | 平台卖什么 | 商品一级分类、在售分类数 | 只介绍实际上架类目 | 引导选择树苗、肥料或园艺用品 |
| 02 | 商品关键词搜索 | 商品名、分类、状态、库存 | 仅返回上架且未删除商品 | 追问商品名或用途 |
| 03 | 按树种找树苗 | 树种标签、品种、库存、价格 | 树种同义词映射 | 追问树种、地区、预算 |
| 04 | 按作物找肥料 | 适用作物、肥料类型、库存 | 适用范围必须来自商品字段 | 追问作物和使用阶段 |
| 05 | 热销商品推荐 | 销量字段、上架状态、库存 | 销量降序；最高销量并列随机选 1 个 | 无数据时改为最新上架商品并明确说明 |
| 06 | 低价商品推荐 | 价格、促销价、库存 | 按实际到手价升序；排除无库存 | 追问预算上限 |
| 07 | 高性价比商品推荐 | 价格、销量、评价或运营定义 | 明确“性价比”计算规则 | 无统一指标时不得宣称性价比最高 |
| 08 | 商品详情 | 商品 ID、详情、规格、图片、库存 | 返回当前实时详情 | 商品不存在或下架时明确提示 |
| 09 | 商品规格/用法 | 规格、单位、适用范围、说明书 | 不得补写未维护的用法 | 跳转商品详情或追问具体商品 |
| 10 | 商品价格/库存 | 当前售价、原价、库存、限购 | 仅返回实时字段 | 价格或库存缺失时提示以商品页为准 |
| 11 | 苹果/梨/桃等果树种植 | 百科树种、种植要点、适宜地区 | 仅引用审核后的百科内容 | 追问地区、季节、苗龄 |
| 12 | 苗木定植流程 | 定植深度、株距、浇水、缓苗知识 | 树种差异必须有知识依据 | 追问树种与地块条件 |
| 13 | 树苗移栽/缓苗 | 移栽季节、根系、浇水知识 | 区分裸根与营养钵苗 | 追问苗木类型和季节 |
| 14 | 浇水问题 | 作物、土壤、季节、症状 | 不给绝对浇水量 | 追问地区、天气、土壤湿度 |
| 15 | 施肥时间 | 作物、树龄、季节、生育期 | 不给缺少作物条件的具体剂量 | 追问作物和当前阶段 |
| 16 | 化肥使用方法 | 商品说明、作物、用量、禁忌 | 商品说明优先于通用知识 | 追问肥料名称、作物、面积 |
| 17 | 有机肥使用 | 产品说明、腐熟要求、施用期 | 明确未腐熟风险 | 追问作物和施用方式 |
| 18 | 复合肥选择 | NPK、适用作物、生育期 | 推荐必须可追溯到商品字段 | 追问作物和生育阶段 |
| 19 | 叶片发黄 | 百科症状、常见原因、风险级别 | 不能将诊断当成结论 | 追问树种、叶片图片、发生位置 |
| 20 | 病虫害识别 | 百科病虫害、图片/症状 | 无图片或依据时不确诊 | 追问图片、树种、部位、时间 |
| 21 | 修剪整形 | 树种、树龄、季节、修剪知识 | 高风险操作须提示专业人员 | 追问树种、树龄、季节 |
| 22 | 开花不结果/长势差 | 百科原因、养护记录 | 列出可能性，不下诊断结论 | 追问树种、树龄、气候、图片 |
| 23 | 查询自己的订单 | 订单 ID、登录用户、订单状态 | 必须鉴权且订单归属当前用户 | 未登录时引导登录；无订单号时追问 |
| 24 | 订单发货/物流 | 订单状态、物流单号、物流状态 | 必须鉴权；物流以实时数据为准 | 订单未发货时说明当前状态 |
| 25 | 订单支付状态 | 订单状态、支付时间、支付方式 | 必须鉴权；不得展示他人订单 | 支付异常转人工/支付页 |
| 26 | 取消订单 | 订单状态、取消资格 | 仅说明流程，不执行取消 | 不符合取消条件时说明原因 |
| 27 | 退款/售后 | 售后状态、退款资格、平台规则 | 仅说明流程，不自动退款 | 未登录或超期时说明入口 |
| 28 | 修改收货信息 | 订单状态、修改资格 | 不执行写操作 | 引导用户使用订单页面或人工客服 |
| 29 | 发票/支付凭证 | 平台发票规则、订单归属 | 必须鉴权；规则以平台配置为准 | 没有能力时明确暂不支持 |
| 30 | 优惠券/活动 | 有效活动、优惠券规则、商品范围 | 不编造活动；以实时配置为准 | 无活动时直接说明 |
| 31 | 商家入驻条件 | 入驻规则、资质要求 | 只展示公开规则 | 引导至入驻入口 |
| 32 | 商家审核进度 | 申请 ID、当前商家身份、审核状态 | 必须鉴权且仅本人申请 | 未申请时引导发起申请 |
| 33 | 商家商品发布 | 商家身份、商品发布规则 | 必须鉴权；不代替发布 | 引导商家后台操作 |
| 34 | 商家订单处理 | 商家身份、订单归属、状态 | 必须鉴权；不直接改订单 | 引导商家后台处理 |
| 35 | 联系人工客服 | 客服电话、在线入口、工作时间 | 仅用平台配置中的真实数据 | 配置缺失时不编造号码 |
| 36 | 投诉/建议反馈 | 反馈入口、问题分类 | 收集必要信息，不承诺处理结果 | 引导提交工单或人工客服 |
| 37 | 模糊问题澄清 | 当前上下文、候选业务域 | 一次只追问一个关键问题 | 提供树苗/肥料/订单/种植四个选项 |
| 38 | 图片或语音无法识别 | 上传能力、图片格式、人工入口 | 不假装看懂图片或语音 | 请求清晰图片或文字描述 |
| 39 | 超出平台范围 | 平台能力边界 | 不提供医疗、法律、金融等专业结论 | 说明能力边界并给出安全建议 |
| 40 | 多次澄清仍无答案 | 对话记录、未解决问题池 | 最多追问 2 次后停止循环 | 安全兜底并写入 DataCenter 未解决问题 |

## 4. 已知 MF_EP 数据/接口缺口

以下能力当前不能仅依靠现有通用关键词搜索稳定完成，MF_EP 负责人必须逐项确认是否已有字段和接口；没有则标记为待实现：

| 缺口 | 所需字段或接口 | 规则归属 |
| --- | --- | --- |
| 热销推荐 | `sales_count` 或统一销量口径；按销量查询接口 | MF_EP |
| 并列随机 | 同销量候选集合与可复现随机规则 | MF_EP 优先，AgentService 可展示结果 |
| 库存过滤 | 上架状态、库存、预售状态的统一定义 | MF_EP |
| 作物/树种匹配 | 商品树种、适用作物、同义词标签 | MF_EP |
| 肥料用法 | 商品说明、适用期、用量、禁忌等结构化字段 | MF_EP |
| 订单售后规则 | 可取消、可退款、可改地址的状态机和公开说明 | MF_EP |
| 活动优惠 | 实时活动、优惠券、商品适用范围查询接口 | MF_EP |

## 5. 验收标准

MF_EP 负责人提交后，按以下标准验收：

1. 40 类均有状态标记，且前 20 类优先标为 `已确认` 或明确列为 `待补数据`。
2. 每类都有真实数据来源，禁止写“由 AI 判断”代替字段或规则。
3. 所有订单、商家审核、售后类均明确鉴权与归属校验。
4. 所有商品推荐类均明确上架、库存、排序和并列规则。
5. 所有模糊问题均明确一次追问内容、最多追问次数和未解决问题去向。
6. YAML 草案与本任务书一致，且无密钥、账号、密码和真实用户隐私数据。
