## 1. 项目定位

`MF_EP` 是苗丰项目的主业务系统，负责商品、苗木百科、科普文章、订单、用户、商家及三端页面。

它是正式业务内容的事实源：DataCenter 负责运营与治理，AgentService 负责智能客服编排和检索，二者都不应绕过 MF_EP 直接写正式电商内容。

## 4. 当前联调状态

| 调用方向 | 当前状态 | 说明 |
| --- | --- | --- |
| MF_EP 客户端 -> MF_AgentService | 已打通 | 客户端开发服务使用 `5174`；管理端 `5173` 不代理 Agent 接口。 |
| MF_AgentService -> MF_EP | 已打通 | Agent 通过 HTTP 查询商品、百科、文章、FAQ 和订单，不直接连接 MF_EP 数据库。 |
| MF_AgentService -> MF_DataCenter | 已具备基础接入 | 咨询记录、工具调用、未解决问题和样本候选可沉淀到 DataCenter。 |
| MF_DataCenter -> MF_EP AI 草稿接口 | 已具备接口落点 | DataCenter 已有创建草稿、发布、下线的调用封装，需继续做真实闭环验收。 |
| MF_EP 发布 -> MF_AgentService 增量同步 | 未打通 | 当前没有发布事件、消息队列或主动同步调用。 |

当前 AgentService 会在启动时和按默认 10 分钟间隔从 MF_EP 拉取公开知识。它不是发布后的即时增量同步，因此新内容不能保证立即被 Agent 检索。

## 6. 未完成项与风险

### P0：跨项目知识闭环

- MF_EP 发布、下线和回滚后会写入待确认同步事件；AgentService 尚未实现事件拉取后的本地索引刷新与确认回写。
- AgentService 当前主要依赖定时拉取；运行时向量 RAG 未启用，不能保证发布后立即检索。
- 尚无 AgentService 侧同步失败重试、检索结果回写和告警。

### P0：版本回滚与高风险人工复核

- 已新增草稿版本快照、按版本回滚和回滚同步事件；执行数据库迁移后可用。
- 高风险草稿默认进入 `pending_review`，仅 `approved` 的人工审核结论可以发布。
- 已记录审核人、审核意见和审核时间；后续仍需由 DataCenter 接入真实权限和审核页面。

### P1：三方自动化验收

- 当前没有覆盖 `DataCenter -> MF_EP 发布 -> AgentService 同步 -> 用户提问命中新内容` 的端到端测试。
- DataCenter 的知识工作台测试主要使用 Mock；MF_EP 的 AI 草稿发布、下线和高风险拦截缺少专项测试。
- 需要建立可重复执行的隔离测试数据和清理机制，避免依赖人工点击或污染正式内容。

### P1：内容与检索质量

- 补充苗木养护、病虫害、肥料使用等高频知识。
- 优化商品关键词、标签和适用场景数据，提升商品推荐命中。
- 针对低置信度与未解决问题，形成 DataCenter 问题池到正式内容的运营闭环。

## 7. 下一阶段建议与责任归属

| 任务 | 主负责人 | 协作方 | 验收标准 |
| --- | --- | --- | --- |
| 正式内容版本、回滚、审核状态机、发布事件 | MF_EP | DataCenter | 可发布、下线、按版本恢复，全程有审计。 |
| 增量同步消费、索引刷新、幂等与失败重试 | MF_AgentService | MF_EP | 新发布内容在约定时限内可被检索；下线后不可命中。 |
| 审核与知识运营工作台 | MF_DataCenter | MF_EP | 候选来源、审核意见、风险等级、发布与同步状态可追溯。 |
| 三方契约和端到端测试 | MF_EP 主导 | DataCenter、AgentService | 自动覆盖发布、下线、高风险拦截、审核放行、回滚、同步失败重试。 |

MF_EP 应作为这项工作的业务主负责人，因为它拥有正式内容、发布权限和回滚责任；DataCenter 不应直接写正式内容，AgentService 不应决定内容是否可上线。
